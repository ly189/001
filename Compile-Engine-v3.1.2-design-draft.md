# 0. 初衷锚定（v3.1.2 必须最优先遵守）

> **Compile-Engine 系列的最终交付物 = Prompt。**
> 不是商业建议、不是探索地图、不是思维报告。

一切机制都必须服务于：

```
识别真实需求 → 生成高质量 Prompt → 帮用户与执行 LLM 高效对话
```

## 锚定原则

1. 任何机制如果让用户**停留在 v3.1.x 内消费输出**而不前往执行 LLM —— 即脱离初衷
2. Possibility Space / Working Hypothesis / Possibility Quality Constraint —— 全部是**中间步骤**，不是终点
3. 系统输出的密度和质量必须**精确够用**，过强的内容质量反而是反向负优化（用户停留消费）

## 核心自检（v3.1.2 PS 输出前必查）

> **用户在选定该方向后，是否能基于此方向生成一个具体可执行的 Prompt？**

- 是 → 通过
- 否（方向过于抽象，无法直接 compile 为 Prompt） → 必须继续暴露下一层 PS，直到方向能落到 Prompt 层面

## 反模式（禁止）

- ❌ PS 给出的是商业战略建议，离 Prompt 太远
- ❌ 用户选定方向后仍无法直接 compile 为 Prompt
- ❌ 输出过于完整以至于用户不再需要执行 LLM
- ❌ Working Hypothesis 写得像咨询报告

---

# 1. 设计动机

v3.1.1 引入 Possibility Space 后暴露两个潜在缺口：

| 缺口 | 描述 | 风险 |
|---|---|---|
| 时间维度失控 | 用户陷入无限暴露方向的认知瘫痪（Possibility Paralysis）| 永远不收敛 → 永远拿不到 Prompt |
| 空间维度退化 | PS 退化为同质枚举（A/B/C/D 罗列器）| 方向无结构差异 → 选定后仍无法 compile 出高质量 Prompt |

**两者都直接威胁初衷的"生成高质量 Prompt"环节。**

v3.1.2 的两个新机制：
- 探索饱和检测 → 修补时间维度
- 方向多样性约束 → 修补空间维度

# 2. 新增机制 A：探索饱和检测（§3.5 扩展）

## 2.1 触发条件（同时满足）

- 当前在第 ≥3 轮 exploring（首轮 + 至少 2 轮后续探索）
- 满足以下饱和信号（任一）：
  - **a** 用户语言重复（同一类问题反复问，无新维度引入）
  - **b** 决策疲劳信号："还是不知道选哪个 / 都好像可以 / 我有点乱 / 越看越没思路 / 都行 / 不知道选哪个"
  - **c** 系统已暴露的方向数 ≥10（含多轮累计）

## 2.2 触发行为（软建议）

```
我们已经探索了几个主要方向。
如果想推进，可以考虑：

1. 低成本试错：选一个最有感觉的方向，做 1 周轻量验证
2. 维度筛选：用 [时间 / 兴趣 / 资源] 任一维度自筛

当然如果还想继续暴露其他维度，告诉我即可。
```

## 2.3 约束

- 仅触发 **1 次/会话**（用户拒绝后不再建议）
- 不替用户选方向（不出现"我推荐 A"）
- 必须保留"继续探索"出口
- 触发后在 Working Hypothesis 中标记 `[饱和提示已发出]`，避免反复触发

## 2.4 不冲突论证

| 与现有原则的关系 | 是否冲突 |
|---|:-:|
| 避免过早收敛 | ✅ 不冲突——"过早" vs "已饱和"，时机不同 |
| 不替用户决策 | ✅ 不冲突——只建议动作类型，不指定具体方向 |
| 探索友好 | ✅ 强化——给探索一个有意义的终点 |
| **初衷（生成 Prompt）** | ✅ **强化——把停滞用户推回 Prompt 生成轨道** |

# 3. 新增机制 B：方向多样性约束（§4.5 扩展，仅 Strategic 级 PS）

## 3.1 核心要求

每个方向必须满足：
- 与其他方向**至少在一个核心维度**上结构性不同
- **不是同一维度上的轻微变体**

## 3.2 核心维度参考池(任选 3+ 个不同维度切片)

- 业务模式（B2B / B2C / 平台 / 工具 / 服务 / 内容）
- 时间周期（短期变现 / 中期资产 / 长期复利）
- 资源依赖（独立完成 / 需团队 / 需资本 / 需供应链）
- 风险结构（线性回报 / 杠杆 / 不可逆 / 高方差）
- 用户关系（一次性 / 订阅 / 社区 / 渠道）
- 价值类型（节省时间 / 节省钱 / 赚钱 / 学习 / 情感）

## 3.3 关键约束（与初衷锚定对接）

> **核心维度池中所有维度，必须最终落到"对生成 Prompt 有指导意义"的层面。**

每个方向暴露后，必须能完成以下闭环：
```
用户选定方向 X → 系统能基于 X 直接 compile 出可执行 Prompt
```

如果无法闭环 → 该方向粒度过粗 → 需进一步分层暴露。

## 3.4 自检问题（双重检查）

输出前必查两条：

**Q1（结构差异）**：「如果用户分别选了方向 A 和方向 B，他们做的事**结构上**是不同的吗？」
- 是 → 通过
- 否 → 重新切片

**Q2（Prompt 可达性）**：「用户选定每个方向后，是否能基于该方向生成一个具体可执行的 Prompt？」
- 是 → 通过
- 否 → 该方向粒度过粗，需进一步分层

## 3.5 反模式

- ❌ 5 个方向都属于同一维度的渠道/形式变体
  例：写公众号 / 写知乎 / 写 Substack / 写博客 / 写 Newsletter
  → 全部是"图文写作-渠道"维度变体，结构无差异
- ❌ 仅按"工具/形式/平台"单一维度切片
- ❌ 方向粒度过粗以至于用户选定后仍无法 compile（如"做长期资产"——这不是方向，是优化变量）
- ❌ 把 PS 写成商业咨询报告（"市场分析显示……"）

## 3.6 允许

- ✓ 跨多维度的方向集
  例：
  - 写作内容（订阅 / B2C / 独立）
  - 数字产品（一次性买断 / B2C / 独立）
  - 评测站（流量驱动 / B2B / 小团队）
  - Newsletter SaaS（订阅 / B2C / 工具型）

- ✓ Medium 级 PS 不强制此约束（用户可能就是要同维度对比）

## 3.7 自检上限（防重写循环）

- 自检 ≤ 2 次
- 超过仍不通过 → 降级保留原输出 + 在末尾标注 `[多样性自检未完全通过，建议人工复核]`
- 不允许无限重写

# 5. 新增 Test Cases（v3.1.2 专项）

## TC-18｜探索饱和检测触发

- **轮 1 输入**：我想做点什么副业
- **轮 1 应触发**：标准 PS（5 方向）
- **轮 2 输入**：还有别的方向吗
- **轮 2 应触发**：再 PS 5 方向（不饱和）
- **轮 3 输入**：还是不知道选哪个 / 我有点乱
- **轮 3 应触发**：饱和检测软建议（1 次/会话）
- **禁触发**：替用户选方向、强制收敛、反复触发

## TC-19｜方向多样性正例

- **输入**：我想做副业（Strategic 级 PS）
- **应触发**：5 方向，每方向至少跨一个核心维度（业务模式 / 时间周期 / 资源依赖 等）
- **禁触发**：5 方向都是"图文写作渠道变体"等单维度同质枚举

## TC-20｜方向多样性反例（自检拦截）

- **模拟系统初次输出**：A) 写公众号 B) 写知乎 C) 写 Substack D) 写博客 E) 写 Newsletter
- **应触发**：自检 Q1 失败（全部"图文-渠道"维度）→ 重写
- **重写后应触发**：跨维度方向集
- **降级条件**：≥2 次重写仍不通过 → 降级输出 + 标注

## TC-21｜初衷锚定反例（防止 drift）

- **输入**：我想做长期资产（用户给的"方向"过抽象）
- **应触发**：识别"长期资产"是优化变量而非方向 → 不接受为已选方向 → 暴露下一层 PS（在"长期资产"目标下的具体方向）
- **禁触发**：直接基于"长期资产"生成 Prompt（粒度不够）
- **禁触发**：把回复写成商业咨询报告

## TC-22｜不退化保证（v3.1.1 行为保留）

- 重新跑 v3.1.1 测试套件中所有不退化红线 TC（TC-01 / TC-08 / TC-10 / TC-16 / TC-17）
- 必须维持 v3.1.1 评分

# 6. 实施前提条件

不满足以下任一条件 → **不启动 v3.1.2 实施**：

1. v3.1.1 已经过外部 LLM 真实使用 ≥1 周
2. 已收集 ≥10 个真实任务的实际响应数据
3. 实证发现以下至少一个：
   - 用户在 v3.1.1 的 exploring 阶段陷入瘫痪（≥2 起）
   - PS 输出退化为同质枚举（≥3 起）
4. v3.1.2 实施前，本草案的初衷锚定原则 §0 必须再次评审

---

# 7. 拒绝清单（v3.1.2 不会做的事）

- ❌ 不在 PS 中加入"系统推荐"或"加权评分"
- ❌ 不让 PS 输出超过 7 个方向（信息过载）
- ❌ 不把饱和检测做成强制收敛
- ❌ 不引入跨会话状态记忆（突破 = v4，不是 v3.1.x）
- ❌ 不让多样性约束在 Medium 级 PS 强制执行
- ❌ 不把 PS 输出写成完整商业咨询报告
- ❌ 不在 exploring 阶段直接生成最终 Prompt（保持 v3.1.1 阶段约束）

---

# 8. 与 v3.1.1 的关系

```
v3.1   = 执行型场景最优
v3.1.1 = 加 Possibility Space（探索型支持）
v3.1.2 = 加 探索饱和检测 + 方向多样性约束
         （PS 机制的质量与终止保护）
```

v3.1.2 是 v3.1.1 的**质量加固版本**，而非新方向。

---

# 9. 之后的边界

> **v3.1.2 大概率是 v3 系列的终点版本。**

理由：
- 单 Prompt 架构在"探索-收敛平衡"上到 v3.1.2 基本触顶
- 再要进步必须进入 v4：状态化认知系统
  - 跨会话记忆用户探索轨迹
  - 真正的概率推理（贝叶斯更新，非 H/M/L 标记）
  - 工具调用 / 多 Agent 协作

v3.1.x 之后的边际收益急剧递减，硬加机制 = 负优化。
